# 数据库迁移排障

本指南在 WeKnora 启动时数据库迁移失败后，从系统信息页面链接进入。涵盖最常见的失败原因、诊断方法，以及在不丢数据的前提下恢复的方式。

若以下情况都不符合你的场景，可直接跳到底部的[反馈问题](#反馈问题)。

> 中文常见问题排查参见 [常见问题](常见问题.md)。

---

## "迁移失败"的含义

WeKnora 每次启动都会自动运行 `golang-migrate` 迁移。当某次迁移失败时，应用**仍会完成启动**（这样 UI 保持可访问，便于你排查问题），但：

- 失败的迁移会被回滚，数据库停留在上一个版本。该迁移引入的表 / 索引**不会被创建**。
- 依赖这些表的下游功能（Wiki 入库、知识图谱、任务队列……）可能静默地不产出任何结果。
- 系统信息页面会显示部分的数据库版本 + 红色的 "Migration failed" 标签及捕获到的错误。

UI 中看到的缓存错误信息，与启动时日志里 `Database migration failed: ...` 打印的内容一致。最近的容器日志是权威来源——在做任何破坏性操作前先把它复制下来。

## 常见原因

### 1. 缺少 PostgreSQL 扩展

许多迁移依赖扩展（`pg_trgm`、`vector`、`pg_search`），通过 `CREATE EXTENSION IF NOT EXISTS` 创建。**`IF NOT EXISTS` 并不会校验扩展是否真正安装**——它只检查系统目录。如果扩展的共享库缺失，或当前角色没有 `CREATE` 权限，名义上创建扩展的那条迁移语句可能会成功，但后续使用该扩展的迁移（例如构建 `gin_trgm_ops` 索引）就会失败。

**错误中的典型症状**：

```
ERROR: operator class "gin_trgm_ops" does not exist for access method "gin"
ERROR: type "vector" does not exist
ERROR: function ... does not exist
```

**修复**：

```sql
-- 以超级用户（通常是 `postgres`）连接：
\c your_weknora_database
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS vector;       -- 若 RETRIEVE_DRIVER 含 postgres
CREATE EXTENSION IF NOT EXISTS pg_search;    -- 仅 ParadeDB 需要

-- 验证扩展是否真正加载：
SELECT extname, extversion FROM pg_extension WHERE extname IN ('pg_trgm','vector','pg_search');
```

然后重启 WeKnora。下次启动会从失败迁移处继续执行。

如果 `CREATE EXTENSION` 自身报 **"could not open extension control file"** 或 **"permission denied"**，说明 PostgreSQL 服务器上并未安装该扩展——安装对应的 OS 包（例如 `pg_trgm` 对应 `postgresql-contrib`），或切换到预装了该扩展的镜像，再重试。

### 2. 脏迁移状态（dirty）

如果某次迁移在中途崩溃（OOM、容器被杀、网络抖动），`golang-migrate` 会把 schema 标记为该失败版本上的 "dirty"。默认情况下 WeKnora 启动时会尝试自动恢复；如果你用 `AUTO_RECOVER_DIRTY=false` 关掉了它，就会看到：

```
database is in dirty state at version N. ...
```

**修复**——使用内置的辅助命令：

```bash
# 查看当前记录的版本
make migrate-version

# 强制把版本回退到最后一次成功的迁移（即提示信息里的 N - 1）
make migrate-force version=<N-1>

# 重新执行待应用的迁移
make migrate-up
```

之后重启 WeKnora。

或者设置 `AUTO_RECOVER_DIRTY=true`（近期版本的默认值）后直接重启——启动时会自动执行同样的 `force` + 重试。

### 3. 数据库角色权限不足

某些迁移会创建扩展或修改共享目录，需要超级用户或 `CREATEROLE` / `CREATEDB` 权限。错误形如：

```
ERROR: permission denied to create extension "pg_trgm"
ERROR: must be owner of database ...
```

**修复**：给 `DB_USER` 使用的角色授予必要权限，或事先以超级用户预创建扩展 / 对象，然后重启。此时迁移里的 `CREATE EXTENSION IF NOT EXISTS` 会变成空操作。

### 4. `CREATE INDEX` 期间磁盘空间不足

GIN / pgvector 索引可能需要大量临时空间。错误：

```
ERROR: could not extend file ...: No space left on device
ERROR: cannot create temporary tables in transaction
```

**修复**：在 `PGDATA` 所在的卷上释放空间，然后重启。迁移会重试索引构建。

### 5. 手工修改导致的 schema 偏移

如果你此前手工改过表 / 列，而后续迁移期望的是原始结构，就会因类型不匹配而失败。最安全的恢复方式是：把线上 schema 对齐到上一次成功迁移的 `*.up.sql`，再重新执行待应用的迁移。

## 通用诊断清单

1. **读完整错误**：UI 里缓存的信息只受你浏览器滚动范围限制——它就是完整的 `golang-migrate` 错误。容器日志显示同样的内容，并带有栈上下文。
2. **定位失败的迁移**：错误中的版本号（或 `make migrate-version`）指向 `migrations/versioned/` 下的某个文件。打开 `migrations/versioned/<version>_*.up.sql`，找到与错误类型对应的语句（扩展、索引、函数、外键……）。
3. **用 `psql` 手动执行那条失败语句**。这样得到的报错会比迁移封装层的具体得多。
4. **修复根因**（安装扩展、修正权限、释放磁盘……），然后二选一：
   - 重启 WeKnora 让自动恢复重试；**或**
   - 在代码检出目录里执行 `make migrate-up`，在服务进程之外应用迁移。
5. **验证**：系统信息页面此时应显示数据库版本且不再有 "Migration failed" 标签，此前坏掉的功能（Wiki、知识图谱……）应开始产出结果。

## 反馈问题

如果你走完了上面的清单迁移仍然失败，请到以下地址提 issue：

<https://github.com/Tencent/WeKnora/issues/new?template=bug_report.yml>

请包含：

- WeKnora 版本 + commit ID（来自系统信息页面）。
- 系统信息页面（或容器日志）里的完整错误。
- PostgreSQL 版本（`SELECT version();`）及部署方式（原生、ParadeDB、Aurora、阿里云 RDS……）。
- 以下命令的输出：
  ```sql
  SELECT extname, extversion FROM pg_extension;
  ```
- `RETRIEVE_DRIVER`、`AUTO_MIGRATE`、`AUTO_RECOVER_DIRTY` 的任何非默认值。

系统信息页面上的 "Report issue" 链接会预填一段包含捕获错误的正文——点它是最快的路径。
