# 文档分块策略指南

WeKnora 如何在上传文档后、embedding 之前对其进行分块，默认值为何如此设定，以及何时需要调整。

## 为什么分块很重要

检索增强生成（RAG）的工作方式是把文档切成小片后嵌入向量索引，再在查询时取回最相关的片段。文档的切片方式--分块大小、重叠量、切点位置--直接决定检索召回率与 LLM 生成答案的质量。

经验上（Vecta 2026 年 2 月对 50 篇学术论文的基准测试）：在 ~512 tokens、~15% 重叠下做递归切分是最强的单旋钮基线，端到端准确率达 69%，优于语义分块与过度设计的混合方案。WeKnora 以此为基础，并在文档给出结构线索时叠加更智能的策略。

## 自适应三级分块

按知识库设置，通过编辑器的 **Chunking**（分块）侧边栏（或 KB 配置 API 的 `strategy` 字段）配置。

| 策略 | 何时选用 | 做什么 |
|------|----------|--------|
| `auto`（推荐） | 新建 KB 的默认值 | 先对文档画像，再从下方链条中选最强的层级。 |
| `heading` | Markdown 式结构 | 在 `#` / `##` / `###` 边界切分。每个分块在 embedding 时前置一个面包屑上下文头（`# Top > ## Section`）。 |
| `heuristic` | PDF 式结构 | 在换页符（form-feed）、编号章节、多语言章节标记（DE / EN / ZH）、全大写标题、视觉分隔符处切分。 |
| `legacy`（= `recursive`） | 其它情况，或作为兜底 | 纯递归分隔符切分器--最新版本，含优先级递归与重叠上限修复。 |

文档画像器先运行，统计结构信号（Markdown 标题、换页符、各语言章节标记、全大写行、视觉分隔符、空行突发）。auto 策略根据这些计数选择层级链条；校验器会拒绝明显异常的输出（如 heading 切分器产出 200 个单行分块）并回退到下一层级。

## 设置参考

### 核心

| 设置 | 范围 | 默认值 | 适用场景 |
|---------|-------|---------|-----------------|
| **分块大小（Chunk size）** | 100–4000 字符 | 512 | 默认值适用大多数场景。FAQ / 原子问答用 200–400。叙事类文档用 1000–2000。 |
| **分块重叠（Chunk overlap）** | 0–500 字符 | 80（≈15%） | FAQ 与结构化记录用 0。通用文档用 80（默认）。论证型文本（推理跨分块）用 150–200。 |
| **分隔符（Separators）** | 字符串列表 | `["\n\n", "\n", "。", "！", "？", ";", "；"]` | 顺序很重要--切分器先尝试高优先级分隔符，仅当某片段仍超限时才回退到低优先级。 |

### 父子分块（Parent-Child）

两级检索：**子分块**（小，用于向量匹配 embedding）与**父分块**（大，返回给 LLM 作为上下文）。

| 设置 | 范围 | 默认值 | 说明 |
|---------|-------|---------|-------|
| **启用父子分块** | 开关 | 开 | 推荐用于 >10 页的文档。短 FAQ 可关闭以减半存储成本。 |
| **父分块大小** | 512–8192 字符 | 4096（≈1000 EN tokens） | 长上下文 LLM（Claude、GPT-4-Turbo）可更大。4k 上下文的本地 LLM 用 1024–2048。 |
| **子分块大小** | 64–2048 字符 | 384（≈95 EN tokens） | 问答式精确匹配用 128–256。embedder 接受 >1000 tokens 时（E5 / BGE-large）可用 512–1024。 |

### 高级

| 设置 | 范围 | 默认值 | 何时设置 |
|---------|-------|---------|-------------|
| **Token 上限（Token limit）** | 0–8192 | 0（关闭） | 当 embedding 模型 token 上限较小时开启。见下表。 |
| **语言（Languages）** | `de` / `en` / `zh`（多选） | 空（自动检测） | 同质语料可显式设置以收窄启发式匹配模式。 |

#### 各 embedding 模型的 token 上限指南

| Embedder | Token 上限 | 推荐 `tokenLimit` 设置 |
|----------|-------------|----------------------------------|
| OpenAI `text-embedding-3-small/large` | 8191 | **0（保持关闭）** |
| Anthropic Voyage-3 | 32000 | **0** |
| Jina-embeddings-v3 | 8192 | **0** |
| Cohere `embed-multilingual-v3` | 512 | **400** |
| BGE-base / BGE-large / E5-large | 512 | **400** |
| Sentence-Transformer `all-MiniLM-L6-v2` | 256 | **200** |

经验法则：任何 token 上限 > 2000 的现代 embedder 都保持 0。对较小 embedder 设为模型硬上限的 80%，以确保即便 CJK 内容（每字符更密）也能始终装得下。

## 场景预设

| 工作负载 | 策略 | ChunkSize | Overlap | 父子分块 |
|----------|----------|-----------|---------|--------------|
| FAQ / 问答知识库 | `auto`（通常会选 legacy） | 200–400 | 0 | 关 |
| Markdown 文档 / wiki | `auto`（选 heading） | 512 | 80 | 开 |
| 带分页符的 PDF 报告 | `auto`（选 heuristic） | 800–1200 | 100–150 | 开 |
| 长篇叙事（书籍、文章） | `auto`（选 recursive） | 1000–2000 | 150–200 | 开 |
| 代码文档 | `legacy` | 800 | 100 | 可选 |
| 混合语言语料 | `auto`，languages 留空 | 512 | 80 | 开 |
| 表格报告 / CSV 派生 | `legacy` | 400 | 0 | 关 |

## 在 UI 中调试

KB 编辑器的 **Chunking** 侧边栏底部有一个「用样本文本测试」折叠区：

1. 粘贴一段 Markdown / 纯文本片段（最大 64 KB）。
2. 点击 **Run preview**（运行预览）。
3. 面板展示：
   - 选中的策略层级（彩色标签）
   - 被拒绝的层级及原因（如「too many tiny chunks」）
   - 文档画像（标题计数、换页符、章节标记、检测到的语言）
   - 全分块集的大小统计（avg / min / max / stddev）
   - 每个分块的卡片：字符数 + 近似 tokens、位置区间、章节面包屑（若设置）、内容预览

此操作对隔离的 goroutine 切分过程只读运行（5 秒超时）--不写数据库、不调 embedding API。可用于在触发重新上传前对比同一样本的不同配置。

## API

有三个端点可写入分块配置。KB 配置更新端点连接编辑器 UI，使用 **camelCase** 与 `documentSplitting` 信封：

```http
PUT /api/v1/initialization/config/:kbId
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "documentSplitting": {
    "chunkSize": 512,
    "chunkOverlap": 80,
    "separators": ["\n\n", "\n", "。", "！", "？", ";", "；"],
    "strategy": "auto",
    "tokenLimit": 0,
    "languages": ["de", "en"],
    "enableParentChild": true,
    "parentChunkSize": 4096,
    "childChunkSize": 384
  }
}
```

`strategy`、`tokenLimit`、`languages` 字段服务端使用基于指针的 DTO：负载中省略表示「不改」，显式发送空字符串 / 0 / [] 则重置为默认值。

KB CRUD 端点（`POST /api/v1/knowledge-bases`、`PUT /api/v1/knowledge-bases/:id`）接受相同字段，但用 **snake_case** 放在 `chunking_config` 信封下：
`{ "chunking_config": { "chunk_size": 512, "chunk_overlap": 80, "strategy": "auto", "token_limit": 0, "languages": ["de", "en"], ... } }`。

预览端点 `POST /api/v1/chunker/preview` 同样使用 snake_case 形式，并额外接收 `text` 字段作为待切分样本。

## 已知权衡

- **一级 heading 感知分块**会把章节面包屑前置到 embedding 输入，每个分块多耗约 5% tokens，换来结构化文档的分块数减少约 30–50%（存储与查询时净省 tokens）。
- **切换策略不会自动重建索引**已有文档。更改 KB 策略后，需重新上传受影响文件（或通过 UI 触发重建索引）才能应用新分块。
- **PDF 中的 OCR 伪影**（竖排布局被逐字断成单独行）无法由任何切分器修复--这是解析器侧的限制。heuristic 层级仍会保持分块与页面边界对齐，可缓解最坏情况。
- **`recursive` 策略值**在 API 中为完整性而保留，但 UI 中有意隐藏：它与 `legacy` 功能接近，增加第五个下拉选项会稀释「自动 / Markdown / 启发式 / legacy」这一有意义的抉择。
